# 浏览器相关问题
- cors  各种跨域问题
- referer 自动设置的值控制
- cookie Set-Cookie失效，或配置无效

## cors问题
如果你是通过在浏览器地址栏直接输入 URL，或者使用 <a> 标签、window.location.href 进行页面级别的跳转，浏览器遇到 302 重定向是绝对不会报跨域错误的，它会乖乖地跳转到新地址
跨域报错，100% 是因为前端使用了 AJAX (Fetch / XHR / Axios) 来请求这个后端 API 
一般流程：
1、浏览器收到 302 响应和 Location 头。
2、浏览器在底层自动、静默地向 Location 指定的新地址发起新的 GET 请求（这个过程对 JS 代码是透明的）。
3、新地址（跨域的目标服务器）返回了数据。
4、致命一步：浏览器会对这个最终响应的数据进行 CORS 检查。如果目标服务器没有在响应头中返回 Access-Control-Allow-Origin，浏览器就会拦截这个响应，并在控制台抛出 CORS 跨域错误，你的前端 JS 代码只能捕获到一个 Network Error，根本拿不到数据。

> 观察network 请求类型， document/Redirect, fetch/Redirect，如果前端第一次请求后端api用的 fetch,那么后端返回302时, location中的地址，也是沿用fetch
> 所以一开始就不能用fetch

### document (或 document/Redirect)：
触发方式：用户在地址栏输入 URL、点击 <a> 标签、提交 <form> 表单，或执行 window.location.href = '...'。
浏览器行为：这属于整页导航。浏览器认为这是用户主动发起的页面跳转，因此完全豁免 CORS 检查。它可以自由地跟随 302 重定向到任何跨域地址，不会报错。

### fetch (或 xhr)：
触发方式：前端 JavaScript 代码使用了 fetch()、XMLHttpRequest，或者基于它们的库（如 Axios、umi-request 等）。
浏览器行为：这属于异步后台请求。浏览器对此类请求执行极其严格的 CORS 检查。当它跟随 302 重定向到一个跨域地址时，如果那个跨域地址的响应头中没有包含 Access-Control-Allow-Origin: 你的前端域名，浏览器就会直接拦截响应，并在控制台抛出红色的 CORS 错误，你的 JS 代码只会收到一个模糊的 Network Error。

## cookie设置浏览器无效

### 浏览器铁律：IP 地址（如 127.0.0.1, 192.168.x.x）和 localhost 绝对不能设置 Domain 属性！

如果你在本地开发，后端代码却执行了 cookie.setDomain("127.0.0.1") 或 cookie.setDomain("localhost")，浏览器会直接拒绝并报这个错。

### 浏览器铁律：Cookie 的 Domain 必须是当前 URL 的精确匹配**，或者是其父域名。
- 如果前端访问的是 https://api.example.com，后端设置 Domain=example.com ✅ (合法，父域名)
- 如果前端访问的是 https://api.example.com，后端设置 Domain=other.com ❌ (非法，会被拦截)
- 如果前端访问的是 https://example.com，后端设置 Domain=.example.com ✅ (合法，现代浏览器支持)

### 如果你注释掉 cookie.setDomain(...)，浏览器会自动将该 Cookie 的 Domain 默认设置为当前请求的精确主机名。这 100% 符合浏览器的安全规则，永远不会报这个黄色警告。

> 所以本地启动，设置domain为空字符串即可
> 如果浏览器中Cookies中设置成功了cookie，但后端接收到的cookie是旧的，绝大多数是因为浏览器插件造成或前端请求时缓存了之前的cookie数据

## referer控制
出于安全考虑（防止恶意网站伪造来源），前端 JavaScript (如 Axios/Fetch) 对 Referer 的控制权限极其有限：你只能“限制”或“隐藏”它，但绝对不能“伪造”或“自定义”它。
以下是控制 Referer 的 4 种主要机制 例如 (前端 localhost:5173 -> 后端 localhost:8080)

### 浏览器的默认控制机制：Referrer Policy
浏览器根据一套名为 Referrer Policy 的规则来决定发送多少 Referer 信息。现代浏览器（Chrome, Edge, Firefox, Safari）的默认策略是： strict-origin-when-cross-origin
这个默认策略的行为是：
- 同源请求（如 5173 请求 5173）：发送完整的 URL（包含协议、域名、端口、路径和参数）。
- 跨域请求（如 5173 请求 8080）：只发送 Origin（协议+域名+端口），隐藏具体的路径和参数。例如，只会发送 http://localhost:5173/，而不会发送 http://localhost:5173/user/profile?id=1。
HTTPS 降级到 HTTP：完全不发送 Referer。

### 如何在前端主动控制 Referer？
方式 A：在 Fetch API / Axios 中控制（针对特定 AJAX 请求）
```javascript
// 强制发送完整的 URL（包含路径），即使是跨域请求
fetch('http://localhost:8080/api/data', {
  method: 'GET',
  referrerPolicy: 'unsafe-url' // 或 'no-referrer' (完全不发), 'origin' (只发域名)
});
```
> *⚠️ 注意：如果你使用的是 Axios，Axios 底层基于 XMLHttpRequest，不支持直接通过 config 设置 referrerPolicy。你必须改用原生 fetch，或者在创建 Axios 实例时自定义底层的 XMLHttpRequest 行为（非常麻烦，不推荐）。*

方式 B：在 HTML 标签中控制（针对页面跳转、图片、表单）
如果你是通过 <a> 标签跳转或 <form> 提交，可以使用 referrerpolicy 属性：
```html
<!-- 点击此链接时，不发送任何 Referer -->
<a href="http://localhost:8080/logout" referrerpolicy="no-referrer">退出登录</a>

<!-- 强制发送完整 URL -->
<a href="http://localhost:8080/api" referrerpolicy="unsafe-url">访问</a>
```

方式 C：全局 HTML Meta 标签（影响整个 React 页面）
在 React 的 public/index.html 的 <head> 中添加，会覆盖该页面所有请求的默认行为：
```html
<!-- 全局禁止发送 Referer -->
<meta name="referrer" content="no-referrer">

<!-- 全局只发送 Origin (域名)，不发送路径 (推荐用于隐私保护) -->
<meta name="referrer" content="origin">
```

### 后端如何控制前端未来的 Referer？
后端可以通过在 HTTP 响应中返回 Referrer-Policy 响应头，来“命令”浏览器在后续请求中如何发送 Referer。
```plain
response.setHeader("Referrer-Policy", "strict-origin-when-cross-origin");
```
